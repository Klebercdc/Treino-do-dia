from fastapi import FastAPI, HTTPException

from services.exam_ocr.service import extract_payload, health_payload
from services.exam_ocr.schemas import ExtractRequest, ExtractResponse

app = FastAPI(title="kronia_exam_ocr")


@app.get("/")
@app.get("/api/exam_ocr")
def health():
    return health_payload()


@app.post("/", response_model=ExtractResponse)
@app.post("/api/exam_ocr", response_model=ExtractResponse)
def extract(payload: ExtractRequest):
    try:
        result = extract_payload(payload.model_dump())
    except ValueError as error:
        raise HTTPException(status_code=400, detail=str(error))

    return ExtractResponse(**result)

2. DIAGNÓSTICO: POR QUE O VERCEL DETECTA COMO FASTAPI
================================================================================

O Vercel faz autodetecção de framework nesta ordem de prioridade:

  1. Setting explícito no painel da UI (já setado como "fastapi" — causa imediata)
  2. Arquivos presentes no deploy que indicam framework

O ".vercelignore" tem uma regra `*.py` que exclui todos os .py — correto.
Mas DEPOIS tem exceções explícitas que REINCLUEM:

  !api/exam_ocr.py        → FastAPI app com app = FastAPI(...) em api/
  !services/exam_ocr/**   → pasta com o serviço completo
  !requirements.txt       → dependências Python com fastapi na primeira linha

Esses três juntos são exatamente o que o detector de FastAPI procura:
  - requirements.txt com fastapi → "tem dependência FastAPI"
  - api/exam_ocr.py com app = FastAPI() → "tem entrypoint FastAPI"

Conclusão: a detecção não é acidental. Foi causada por uma configuração que
tentou incluir o exam_ocr como função serverless Python do Vercel.

  Contexto: Foi incluído no .vercelignore com o comentário "OCR publicável usado
  pelo pipeline de labs". Isso sugere intenção de deployar como função Vercel.
  Mas o vercel.json só declara "api/*.js" — o exam_ocr.py não tem runtime
  Python declarado no vercel.json, então o Vercel não sabe como executá-lo como
  função serverless mesmo quando incluído.

  Opção A — exam_ocr.py NÃO precisa estar no Vercel (é microserviço local/externo):
    → Remover !api/exam_ocr.py, !services/exam_ocr/**, !requirements.txt do
      .vercelignore
    → Remover requirements.txt da raiz
    → O serviço continua rodando localmente via npm run exam:ocr:dev
    → Impacto: qualquer rota que chame /api/exam_ocr diretamente do deploy
      Vercel vai quebrar. Precisa confirmar se há chamada de produção.

  Opção B — exam_ocr.py PRECISA estar no Vercel como função Python serverless:
    → Adicionar no vercel.json:
        "api/exam_ocr.py": { "runtime": "@vercel/python", "maxDuration": 60 }

    → requirements.txt na raiz: mantém (necessário para @vercel/python)

  Opção C — exam_ocr.py vira função JS/Node (sem Python):
    → Mais trabalho. Não recomendo sem necessidade clara.

NOTA / RISCO IDENTIFICADO — pastor-diagnostico.js
-------------------------------------------------
O arquivo src/server/legacy/pastor-diagnostico.js é um cron Vercel (roteado
via api/system.js:322 -> '__route=pastor-diagnostico'), roda 03:00 UTC.

A função readSupabaseKey() (linhas 39-48) faz FALLBACK para chave anon:

    function readSupabaseKey() {
      return process.env.SUPABASE_SERVICE_KEY
        || process.env.SUPABASE_SERVICE_ROLE_KEY
        || process.env.SUPABASE_ANON_KEY            <-- fallback perigoso
        || process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY <-- fallback perigoso
        || process.env.VITE_SUPABASE_SERVICE_KEY
        || process.env.VITE_SUPABASE_SERVICE_ROLE_KEY
        || process.env.VITE_SUPABASE_ANON_KEY        <-- fallback perigoso
        || '';
    }

Se num ambiente as service keys não estiverem setadas mas a anon estiver,
o cron vai chamar reset_monthly_quotas com chave anon e agora receberá
403 / permission denied (silenciosamente, dependendo do tratamento).

Comparação: src/server/apihelpers/_plans.js (usado pelos demais helpers,
inclusive _diagnosticTracker.js) JÁ tem a versão correta — só lê
SUPABASE_SERVICE_KEY / SUPABASE_SERVICE_ROLE_KEY, sem fallback anon.

AÇÃO PROPOSTA (aguardando ok):

  deixando só as chaves de serviço. Isso evita chamada acidental com anon e

4a) VIEW diagnostic_execution_health — SECURITY DEFINER
-------------------------------------------------------
Definição no repo (2 lugares, ambos como CREATE OR REPLACE VIEW SEM anotação
de security):
  * sql/021_admin_diagnostics_observability.sql:156
  * sql/022_admin_diagnostics_refinements.sql:3
Por padrão, view do Postgres é SECURITY INVOKER. O alerta do linter indica
que no banco REAL a view está como SECURITY DEFINER (provavelmente herdado
do owner na criação).

Efeito do SECURITY DEFINER: a view lê diagnostic_executions com privilégios
do criador (não do chamador). Um authenticated sem policy de SELECT na tabela
base conseguiria ler dados via a view — vazamento potencial.

Efeito de mudar para security_invoker = on: a view passa a respeitar as
policies RLS do chamador nas tabelas base. Já existe a policy
diagnostic_executions_admin_read (to authenticated using
kronia_is_diagnostic_admin()), então admin continua lendo. O único consumidor
de código é src/server/legacy/admin-diagnostics.js:293,544,635 que usa
service_role (bypassa RLS) — NÃO quebra.

Compatibilidade: security_invoker exige Postgres 15+. Projeto roda PG 17. OK.

AÇÃO PROPOSTA (aguardando ok):
  Adicionar "WITH (security_invoker = on)" na definição da view nos arquivos
  sql/021 e sql/022 do repo, para a definição do repo bater com a correção
  desejada no banco.

4b) POLICIES RLS WITH CHECK (true) em diagnostic_executions / diagnostic_steps
------------------------------------------------------------------------------
O REPO JÁ TEM A VERSÃO CORRETA (não é WITH CHECK true):
  sql/021_admin_diagnostics_observability.sql:128-145
    create policy diagnostic_executions_admin_insert
      on public.diagnostic_executions for insert to authenticated
      with check (public.kronia_is_diagnostic_admin());
    create policy diagnostic_steps_admin_insert
      on public.diagnostic_steps for insert to authenticated
      with check (public.kronia_is_diagnostic_admin());

Ou seja, o WITH CHECK (true) é problema só no BANCO real (versão antiga da
policy). O código do repo não contém WITH CHECK (true) para essas tabelas.
(As ocorrências de WITH CHECK (true) em sql/020_exercise_discovery_module.sql
e migration 018 são de OUTRAS tabelas — fora do escopo deste item.)

Quem insere legitimamente nessas tabelas:
  * src/server/apihelpers/_diagnosticTracker.js:364 (diagnostic_executions),
    385 (diagnostic_steps) — via plans.supabaseRequest() que usa service_role
    (ver src/server/apihelpers/_plans.js:46-47: apikey + Bearer com
    SUPABASE_SERVICE_KEY). Bypassa RLS — policy é irrelevante para ele.
  * src/server/legacy/admin-diagnostics.js — só LEITURA, não insere.

AÇÃO PROPOSTA (aguardando ok):
  Re-executar sql/021_admin_diagnostics_observability.sql no banco. É
  idempotente (drop policy if exists antes de create). Substitui a policy
  permissiva WITH CHECK (true) pela restrita kronia_is_diagnostic_admin().
  Não quebra o tracker (service_role bypassa RLS).

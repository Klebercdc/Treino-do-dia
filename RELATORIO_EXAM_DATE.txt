PROBLEMA
--------
O KRONOS ordenava os exames de sangue pela data de upload (processed_at),
não pela data real do documento. Se o usuário enviasse hoje um exame de
6 meses atrás, o KRONOS o tratava como o mais recente, distorcendo a
análise longitudinal de evolução dos biomarcadores.


SOLUÇÃO
-------
Extrair a data real do exame durante o OCR e persistir como campo
estruturado (exam_date DATE) na tabela lab_reports. A ordenação do
histórico passa a usar essa data, com fallback para processed_at
nos exames já existentes.


================================================================================
MIGRATION — APLICAR NO SUPABASE
================================================================================

Arquivo: supabase/migrations/20260603120000_lab_reports_exam_date.sql

---- INÍCIO DO SQL ----

-- Adiciona coluna exam_date em lab_reports
-- Populada pelo pipeline OCR com a data real do exame extraída do documento.
-- Exames anteriores ficam NULL e aparecem por último na ordenação (NULLS LAST).

ALTER TABLE lab_reports
  ADD COLUMN IF NOT EXISTS exam_date DATE;

-- Índice para ordenação eficiente por usuário + data do exame
CREATE INDEX IF NOT EXISTS idx_lab_reports_exam_date
  ON lab_reports (user_id, exam_date DESC NULLS LAST);

---- FIM DO SQL ----

ROLLBACK (se necessário)
================================================================================

A coluna exam_date é nullable e o índice é separado.
Para reverter sem quebrar nada:

  DROP INDEX IF EXISTS idx_lab_reports_exam_date;
  ALTER TABLE lab_reports DROP COLUMN IF EXISTS exam_date;

O código voltaria a usar processed_at como antes (comportamento anterior).

------------------------------------------------------------
A API /api/kronia/exercises/details retornou erro 503 em
100% das requisições após o deploy do commit fd5788d.

Resposta observada pelo usuário:

    HTTP 503
    {
      "type": "exercise_error",
      "code": "SERVICE_UNAVAILABLE",
      "message": "Serviço de exercícios temporariamente indisponível."
    }

A falha NÃO estava relacionada a:
  - Supabase / banco de dados
  - Vídeos ou arquivos de mídia
  - Dados de exercícios em si
  - Rate limit ou sobrecarga de rede

A causa raiz foi um erro de runtime por incompatibilidade entre
ESM (ES Modules) e CommonJS no arquivo:

    src/lib/exercises/local-catalog.ts

O erro ocorria no carregamento do módulo (antes da execução de
qualquer handler), quebrando toda a rota antes que qualquer
try/catch pudesse capturá-lo.

4. CAUSA RAIZ
------------------------------------------------------------

4.1  Uso de import.meta.url em contexto CommonJS

Em sessão anterior, ao tentar criar um fallback de leitura
resiliente para o ambiente serverless da Vercel, foi adicionado
o seguinte bloco em local-catalog.ts (linhas 158–165):

    try {
      raw = JSON.parse(fs.readFileSync(LOCAL_CATALOG_FILE, 'utf8'));
    } catch {
      try {
        const alt = path.join(
          path.dirname(new URL(import.meta.url).pathname),  // ← BUG
          '../../../data/exercises.json'
        );
        raw = JSON.parse(fs.readFileSync(alt, 'utf8'));
      } catch {
        cachedIndex = [];
        return cachedIndex;
      }
    }

  import.meta.url é sintaxe válida APENAS em módulos ES (ESM).

  O tsconfig.json declara "module": "node16". Em funções
  serverless Next.js (src/app/api/**), o TypeScript compila para
  CommonJS — onde import.meta não existe.

  O TypeScript emitiu o erro TS1470, porém, sem "noEmitOnError"
  no tsconfig, continuou gerando o arquivo .js com a sintaxe ESM
  embutida.

  Quando o Node.js 22 (serverless Vercel) tentou carregar esse
  módulo via require(), detectou a presença de import.meta e
  rejeitou o módulo inteiro.

4.2  Tipo incorreto em LocalCatalogMatch.gifUrl

  A interface declarava:

    export type LocalCatalogMatch = {
      gifUrl: string;   // ← não aceita null
      ...
    };

  Após refatoração que removeu o filtro .filter(item => item.gifUrl),
  o campo record.gifUrl passou a poder ser null (quando
  sanitizeMediaUrl retorna null para hosts inválidos como
  cdn.treino-do-dia.app).

  A atribuição gifUrl: record.gifUrl violava o contrato de tipos,
  gerando TS2322 silenciosamente em produção.

4.3  Build inconsistente — ausência de noEmitOnError

  Sem "noEmitOnError": true no tsconfig, o compilador TypeScript
  emitia arquivos .js mesmo na presença de erros. Código inválido
  chegou à produção sem nenhum bloqueio de pipeline.

4.4  Por que funcionava localmente e quebrava na Vercel

  Em desenvolvimento, o tsx (usado para executar TypeScript
  diretamente) suporta import.meta nativamente mesmo em arquivos
  que seriam compilados para CommonJS. A divergência entre o
  ambiente de execução local e o ambiente de compilação da Vercel
  não foi detectada antes do deploy.

6.2  Correção do tipo LocalCatalogMatch.gifUrl (FIX — P1)

ANTES:

      gifUrl: string;       // ← não aceitava null

6.3  noEmitOnError no tsconfig.json (HARDENING — P1)
─────────────────────────────────────────────────────────────

ANTES:
    "compilerOptions": {
      "strict": true,
      ...
    }

      "strict": true,

o build da Vercel vai interromper o processo antes do deploy —
impedindo que código inválido chegue à produção silenciosamente.

9. LIÇÕES APRENDIDAS
------------------------------------------------------------

1. import.meta.url é armadilha clássica em projetos híbridos
   TypeScript com target CommonJS. O tsx mascara o problema
   localmente; o compilador o expõe apenas na Vercel.

2. Sem noEmitOnError, o TypeScript é conivente: emite código
   inválido silenciosamente. Build verde ≠ código correto.

3. Fallback de resiliência que usa API incompatível com o
   ambiente de destino é pior que não ter fallback — cria
   uma falha total onde antes havia degradação parcial.

4. O ambiente de teste local deve espelhar o ambiente de
   compilação de produção. Divergências entre tsx (dev) e
   tsc (prod) precisam ser monitoradas.

3. COMO O BACKFILL DE exam_date FOI FEITO
------------------------------------------------------------

Estado inicial: 3 exames (lab_reports), TODOS com exam_date NULL.
Nenhum tinha data estruturada em normalized_payload, ai_insights
ou metadata. A data só existia no raw_text do OCR.

Auditoria de segurança (por que NÃO usar regex ingênuo):
  - Nos laudos UNIMED, o OCR agrupa os rótulos antes dos valores.
    A primeira data após "Data Entrada" no texto é, na verdade, a
    DATA DE NASCIMENTO (25/02/1985). Um regex automático pegaria a
    data errada. Isso ficou comprovado durante a auditoria.

Datas aplicadas (fonte documentada por exame):
  - faeec9e6 -> 2024-01-12
    Fonte: "Coleta: 12/01/2024" (rótulo explícito e inequívoco).
  - 04dfcb1e (UNIMED) -> 2026-02-25
    Fonte: Data de Entrada/coleta 25/02/2026, coerente com cadeia
    do laudo (assinatura eletrônica 27/02/2026, Data de Saída
    03/03/2026) e DISTINTA da Data de Nascimento 25/02/1985.
  - 51310243 (UNIMED) -> 2026-02-25
    Fonte: mesmo atendimento/laboratório, mesma Data de Entrada.

Forma de aplicação:
  - UPDATEs idempotentes (WHERE id = X AND exam_date IS NULL),
    aplicados via conexão administrativa ao Supabase.
  - Nenhuma data foi inventada: todas vieram do laudo.

Resultado (ordenado por exam_date DESC NULLS LAST):
  04dfcb1e  2026-02-25
  51310243  2026-02-25
  faeec9e6  2024-01-12
  -> 0 exames com exam_date NULL.

Validação do efeito: o exame faeec9e6 foi enviado (upload) em
2026-05-30, mas a coleta foi em 2024-01-12. Agora ele aparece por
ÚLTIMO na ordenação por data real — provando que a lista segue a
data do exame, não a do upload.

Script idempotente (scripts/backfill-lab-exam-date.mjs):
  - Para uso futuro/qualquer ambiente.
  - Extração CONSERVADORA: só preenche quando há uma única data de
    "Coleta" inequívoca no raw_text; layouts ambíguos ficam NULL e
    são listados para revisão manual.
  - dry-run por padrão; grava só com --apply.
  - Requer SUPABASE_URL e SUPABASE_SERVICE_ROLE_KEY no ambiente.
